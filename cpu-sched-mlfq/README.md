```
提示符> ./mlfq.py
```


该轨迹以毫秒为单位精确显示了调度器的决策。在此示例中，它首先运行作业 0 7 毫秒，直到作业 0 发出 I/O；这完全可预测，因为作业 0 的 I/O 频率设置为 7 毫秒，意味着每运行 7 毫秒，它将发出一个 I/O 并等待其完成后再继续。此时，调度器切换到作业 1，该作业仅运行 2 毫秒后发出 I/O。调度器以这种方式打印整个执行轨迹，最后还计算每个作业的响应时间和周转时间以及平均值。

你还可以控制模拟的其他各个方面。例如，你可以指定系统中希望有多少个队列 (-n)，以及所有这些队列的时间片长度应该是多少 (-q)；如果你想要更多控制并为每个队列设置不同的时间片长度，你可以使用 -Q 来指定每个队列的时间片长度，例如，-Q 10,20,30 模拟一个具有三个队列的调度器，最高优先级队列具有 10 毫秒时间片，次高优先级队列具有 20 毫秒时间片，低优先级队列具有 30 毫秒时间片。

你也可以分别控制每个队列的时间分配额。这可以通过 -a 为所有队列设置，或通过 -A 为每个队列单独设置，例如，-A 20,40,60 将每个队列的时间分配额分别设置为 20 毫秒、40 毫秒和 60 毫秒。请注意，虽然本章讨论分配额时指的是时间，但此处是指时间片数量，即，如果给定队列的时间片长度为 10 毫秒，分配额为 2，作业在该队列级别可以运行 2 个时间片（20 毫秒）后才降低优先级。

如果你是随机生成作业，你还可以控制它们可能运行多长时间 (-m)，或者它们生成 I/O 的频率 (-M)。但是，如果你希望对系统中运行的作业的确切特征有更多控制，可以使用 -l（小写 L）或 --jlist，它允许你指定希望模拟的确切作业集合。列表格式为：x1,y1,z1:x2,y2,z2:... 其中 x 是作业的开始时间，y 是运行时间（即，它需要的 CPU 时间），z 是 I/O 频率（即，运行 z 毫秒后，作业发出 I/O；如果 z 为 0，则不发出 I/O）。

例如，如果你想重现图 8.3 中的示例，你将按如下方式指定作业列表：

```
提示符> ./mlfq.py --jlist 0,180,0:100,20,0 -q 10
```

以这种方式运行模拟器会创建一个三级 MLFQ，每个级别具有 10 毫秒时间片。创建了两个作业：作业 0 在时间 0 开始，总共运行 180 毫秒，并且从不发出 I/O；作业 1 在 100 毫秒开始，只需要 20 毫秒的 CPU 时间来完成，并且也从不同发出 I/O。

最后，还有三个有趣的参数。-B 标志如果设置为非零值，则每 N 毫秒将所有作业提升到最高优先级队列，调用方式如下：

```
提示符> ./mlfq.py -B N
```

调度器使用此功能来避免如章节中讨论的饥饿问题。但默认情况下它是关闭的。

-S 标志调用较旧的规则 4a 和 4b，这意味着如果作业在完成其时间片之前发出 I/O，它将在恢复执行时返回到相同的优先级队列，并保持其完整的分配额。这可能导致对调度器的博弈。

最后，你可以通过使用 -i 标志轻松更改 I/O 的持续时间。在此简单模型中，默认情况下每个 I/O 需要固定的 5 毫秒时间，或者你可以通过此标志将其设置为其他值。

你还可以通过 -I 标志来尝试完成 I/O 的作业是移动到其所在队列的头部还是尾部。试一试，很有趣！
