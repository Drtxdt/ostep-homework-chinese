# 概述

这份作业让你探索一些实际代码，这些代码使用锁和条件变量来实现本章讨论的各种形式的生产者/消费者队列。你将查看实际代码，以不同配置运行它，并用它来了解哪些方法有效、哪些无效，以及其他复杂细节。

代码的不同版本对应了"解决"生产者/消费者问题的不同方法。大多数版本不正确；只有一个版本正确。阅读本章以了解更多关于生产者/消费者问题是什么，以及代码通常做什么。

第一步是下载代码并输入 `make`来构建所有变体。你应该看到四个：

- `main-one-cv-while.c`: 使用单个条件变量解决生产者/消费者问题。
- `main-two-cvs-if.c`: 相同，但使用两个条件变量，并使用 `if`检查是否睡眠。
- `main-two-cvs-while.c`: 相同，但使用两个条件变量，并使用 `while`检查是否睡眠。这是正确的版本。
- `main-two-cvs-while-extra-unlock.c`: 相同，但在 `fill`和 `get`例程周围释放锁然后重新获取锁。

查看 `pc-header.h`也很有用，它包含了所有这些不同主程序的通用代码，以及 Makefile 以正确构建代码。

每个程序接受以下标志：

- `-l <每个生产者生产的项目数量>`
- `-m <共享生产者/消费者缓冲区的大小>`
- `-p <生产者数量>`
- `-c <消费者数量>`
- `-P <睡眠字符串：生产者在各点应如何睡眠>`
- `-C <睡眠字符串：消费者在各点应如何睡眠>`
- `-v [详细标志：跟踪并打印正在发生的情况]`
- `-t [计时标志：计时整个执行过程并打印总时间]`

前四个参数相对自解释：`-l`指定每个生产者应循环多少次（从而每个生产者生产多少数据项），`-m`控制共享缓冲区的大小（大于或等于一），`-p`和 `-c`分别设置有多少生产者和消费者。

更有趣的是两个睡眠字符串，一个用于生产者，一个用于消费者。这些标志允许你在执行中的某些点使每个线程睡眠，从而切换到其他线程；这样做可以让你试验每个解决方案，或许能找出特定问题，或研究生产者/消费者问题的其他方面。

字符串的指定方式如下。例如，如果有三个生产者，字符串应单独指定每个生产者的睡眠时间，用冒号分隔。这三个生产者的睡眠字符串可能如下所示：

```
生产者0的睡眠字符串:生产者1的睡眠字符串:生产者2的睡眠字符串
```

反过来，每个睡眠字符串是一个逗号分隔的列表，用于决定在代码中的每个睡眠点应睡眠多少。为了更好地理解这个每个生产者或每个消费者的睡眠字符串，让我们查看 main-two-cvs-while.c 中的代码，特别是生产者代码。在这个代码片段中，一个生产者线程循环一段时间，通过调用 `do_fill()`将元素放入共享缓冲区。在这个填充例程周围有一些等待和发信号的代码，以确保当生产者尝试填充时缓冲区未满。更多细节请参见章节。

```
void *producer(void *arg) {
    int id = (int) arg;
    int i;
    for (i = 0; i < loops; i++) {   p0;
        Mutex_lock(&m);             p1;
        while (num_full == max) {   p2;
            Cond_wait(&empty, &m);  p3;
        }
        do_fill(i);                 p4;
        Cond_signal(&fill);         p5;
        Mutex_unlock(&m);           p6;
    }
    return NULL;
}
```

从代码中可以看出，有一些标记为 p0、p1 等的点。这些点是代码可以被置入睡眠的地方。消费者代码（此处未显示）也有类似的等待点（c0 等）。

为生产者指定睡眠字符串很容易：只需确定生产者在每个点 p0、p1、...、p6 应睡眠多长时间。例如，字符串 `1,0,0,0,0,0,0`指定生产者应在标记 p0 处（在获取锁之前）睡眠 1 秒，然后每次循环中不再睡眠。

现在让我们展示运行其中一个程序的输出。首先，假设用户仅用一个生产者和一个消费者运行。让我们不添加任何睡眠（这是默认行为）。在这个例子中，缓冲区大小设置为 2（-m 2）。

首先，构建代码：

```
prompt> make
gcc -o main-two-cvs-while main-two-cvs-while.c -Wall -pthread
gcc -o main-two-cvs-if main-two-cvs-if.c -Wall -pthread
gcc -o main-one-cv-while main-one-cv-while.c -Wall -pthread
gcc -o main-two-cvs-while-extra-unlock main-two-cvs-while-extra-unlock.c -Wall -pthread
```

现在我们可以运行一些东西：

```
prompt> ./main-two-cvs-while -l 3 -m 2 -p 1 -c 1 -v
```

在描述这个简单例子中发生了什么之前，让我们先理解输出中共享缓冲区的描绘，如左侧所示。起初它是空的（一个空槽用 `---`表示，两个空槽显示为 `[*--- --- ]`）；输出还显示缓冲区中的条目数 (`num_full`)，它从 0 开始。然后，在 P0 放入一个值后，其状态变为 `[u 0 f--- ]`（且 `num_full`变为 1）。你可能还会注意到这里有一些额外的标记；u 标记显示 `use_ptr`的位置（这是下一个消费者将从中获取值的地方）；类似地，f 标记显示 `fill_ptr`的位置（这是下一个生产者将生产值的地方）。当你看到 `*`标记时，它仅表示 `use_ptr`和 `fill_ptr`指向同一个槽。

右侧你可以看到每个生产者和消费者即将执行哪个步骤的跟踪。例如，生产者获取锁 (`p1`)，然后，因为缓冲区有一个空槽，向其中生产一个值 (`p4`)。然后它继续直到释放锁的点 (`p6`)，然后尝试重新获取锁。在这个例子中，消费者获取了锁并消费了值 (`c1, c4`)。进一步研究跟踪以理解当只有一个生产者和一个消费者时生产者/消费者解决方案如何工作。

现在让我们添加一些暂停来改变跟踪的行为。在这种情况下，假设我们想让生产者睡眠，以便消费者可以先运行。我们可以按如下方式实现：

```
prompt> ./main-two-cvs-while -l 1 -m 2 -p 1 -c 1 -P 1,0,0,0,0,0,0 -C 0 -v

结果：

  0 [*---  --- ] p0

  0 [*---  --- ] p1
  1 [u  0 f--- ] p4

```

如你所见，生产者命中代码中的 `p0`标记，然后从其睡眠规范中获取第一个值，本例中为 1，因此每个生产者在甚至尝试获取锁之前睡眠 1 秒。因此，消费者得以运行，获取锁，但发现队列为空，因此睡眠（释放锁）。然后生产者（最终）运行，所有操作如你所料进行。

请注意：必须为每个生产者和消费者指定睡眠规范。因此，如果你创建两个生产者和三个消费者（使用 `-p 2 -c 3`），你必须为每个指定睡眠字符串（例如 `-P 0:1`或 `-C 0,1,2:0:3,3,3,1,1,1`）。睡眠字符串可以比代码中的睡眠点数量短；剩余的睡眠槽被初始化为零。
