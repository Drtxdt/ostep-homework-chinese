# 概述

此程序 `afs.py`允许你体验安德鲁文件系统 (AFS) 的缓存一致性行为。程序生成随机的客户端操作序列（包括文件的打开、读取、写入和关闭），让用户可以预测各个文件最终的值。

以下是一个示例运行：

```
提示符> ./afs.py -C 2 -n 1 -s 12

      服务器                         客户端0                         客户端1
文件:a 包含:0
                             打开:a [文件描述符:0]
                             写入:0 值? -> 1
                             关闭:0
                                                          打开:a [文件描述符:0]
                                                          读取:0 -> 值?
                                                          关闭:0
文件:a 包含:?
提示符>
```

操作序列相对容易阅读。左侧是服务器，每一列显示了两个客户端各自执行的操作（使用 `-C <客户端数量>`指定不同的客户端数量）。每个客户端生成一个随机操作 (`-n 1`)，操作可以是打开/读取/关闭一个文件，或是打开/写入/关闭一个文件。为了简单起见，文件的内容始终只是一个数字。

要生成不同的操作序列，请使用 `-s`设置随机种子，就像往常一样。这里我们将其设置为 12 以获取此特定序列。

在序列中，服务器显示了系统中所有文件的初始内容：

```
文件:a 包含:0
```

如你所见，此序列中只有一个文件 (a)，其值为 0。

时间向下递增，接下来是客户端 0 (c0) 打开文件 'a'（返回一个文件描述符，本例中为 0），向该描述符写入，然后关闭文件。

你马上会看到向你提出的第一个问题：

```
写入:0 值? -> 1
```

当向描述符 0 写入时，你正在用新值 1 覆盖一个旧值。那个旧值是多少？（本例中很简单：0）。

然后客户端 1 开始执行一些操作 (c1)。它打开文件，读取它，然后关闭它。我们再次有一个问题需要回答：

```
读取:0 -> 值?
```

当读取此文件时，客户端 1 应该看到什么值？同样，根据 AFS 的一致性，答案是显而易见的：1（c0 关闭文件并更新服务器时放入文件的值）。

序列中的最后一个问题是服务器上文件的最终值：

```
文件:a 包含:?
```

同样，这里的答案很简单：1（由 c0 生成）。

要查看你是否正确回答了这些问题，请使用 `-c`标志（或 `--compute`）运行，如下所示：

```
提示符> ./afs.py -C 2 -n 1 -s 12 -c

      服务器                         客户端0                         客户端1
文件:a 包含:0
                             打开:a [文件描述符:0]
                             写入:0 0 -> 1
                             关闭:0
                                                          打开:a [文件描述符:0]
                                                          读取:0 -> 1
                                                          关闭:0
文件:a 包含:1
提示符>
```

从该序列中，你可以看到所有问号都已填入答案。

还可以通过 '-d' ('--detail') 标志获取有关发生了什么的更多细节。以下示例显示了每个客户端何时向服务器发出获取或放置文件的操作：

```
提示符> ./afs.py -C 2 -n 1 -s 12 -c -d 1

      服务器                         客户端0                         客户端1
文件:a 包含:0
                             打开:a [文件描述符:0]
获取文件:a 客户端:c0 [0]

                             写入:0 0 -> 1

                             关闭:0
放置文件:a 客户端:c0 [1]

                                                          打开:a [文件描述符:0]
获取文件:a 客户端:c1 [1]

                                                          读取:0 -> 1

                                                          关闭:0

文件:a 包含:1
提示符>
```

你可以通过更高级别的详细信息显示更多内容，包括缓存失效、每一步之后确切的客户端缓存状态以及额外的诊断信息。我们将在下面的另一个示例中展示这些。

随机客户端操作对于生成新问题和尝试解决它们很有用；但是，在某些情况下，为了查看特定的 AFS 行为，精确控制每个客户端的操作是很有用的。为此，你可以使用 `-A`和 `-S`标志（可以一起或单独使用）。

`-S`标志允许你控制客户端操作的确切调度。以上述示例为例。假设我们希望首先完整运行客户端 1；为了实现这个目标，我们只需运行以下命令：

```
提示符> ./afs.py -C 2 -n 1 -s 12 -S 111000

      服务器                         客户端0                         客户端1
文件:a 包含:0
                                                          打开:a [文件描述符:0]
                                                          读取:0 -> 值?
                                                          关闭:0
                             打开:a [文件描述符:0]
                             写入:0 值? -> 1
                             关闭:0
文件:a 包含:?
提示符>
```

这里的 -S 标志传递了 "111000"，意思是"先运行客户端 1，然后客户端 1，然后再次 1，然后 0，0，0，然后重复（如果需要）"。在这种情况下，结果是客户端 1 在客户端 1 写入文件 a 之前读取了它。

`-A`标志精确控制客户端执行哪些操作。以下是一个示例：

```
提示符> ./afs.py -s 12 -S 011100 -A oa1:r1:c1,oa1:w1:c1

      服务器                         客户端0                         客户端1
文件:a 包含:0
                             打开:a [文件描述符:1]
                                                          打开:a [文件描述符:1]
                                                          写入:1 值? -> 1
                                                          关闭:1
                             读取:1 -> 值?
                             关闭:1
文件:a 包含:?
提示符>
```

在此示例中，我们通过 `-A oa1:r1:c1,oa1:w1:c1`指定了以下内容。该列表用逗号分隔每个客户端的操作；因此，客户端 0 应执行 `oa1:r1:c1`所指示的操作，而客户端 1 应执行字符串 `oa1:w1:c1`所指示的操作。解析每个命令字符串很简单：`oa1`表示打开文件 'a' 并为其分配文件描述符 1；`r1`或 `w1`表示读取或写入文件描述符 1；`c1`表示关闭文件描述符 1。

那么客户端 0 的读取将返回什么值？

我们还可以通过一些额外的标志 (`-d 7`) 查看缓存状态、回调和失效：

```
提示符> ./afs.py -s 12 -S 011100 -A oa1:r1:c1,oa1:w1:c1 -c -d 7

      服务器                         客户端0                         客户端1
文件:a 包含:0
                             打开:a [文件描述符:1]
获取文件:a 客户端:c0 [0]
                             [a: 0 (v=1,d=0,r=1)]

                                                          打开:a [文件描述符:1]
获取文件:a 客户端:c1 [0]
                                                           [a: 0 (v=1,d=0,r=1)]

                                                          写入:1 0 -> 1
                                                           [a: 1 (v=1,d=1,r=1)]

                                                          关闭:1
放置文件:a 客户端:c1 [1]
回调: 客户端:c0 文件:a
                             使 a 失效
                             [a: 0 (v=0,d=0,r=1)]
                                                           [a: 1 (v=1,d=0,r=0)]

                             读取:1 -> 0
                             [a: 0 (v=0,d=0,r=1)]

                             关闭:1

文件:a 包含:1
提示符>
```

从此序列中，我们可以看到当客户端 1 关闭（修改后的）文件时会发生什么。此时，c1 将文件放置到服务器。服务器知道 c0 缓存了该文件，因此向 c0 发送失效消息。但是，c0 已经打开了该文件；结果，缓存将保留旧内容，直到文件关闭。

你可以在整个序列中跟踪缓存内容（通过正确的 `-d`标志可用，特别是任何将第三低有效位设置为 1 的值，例如 `-d 4, -d 5, -d 6, -d 7`等）。当客户端 0 打开文件时，你可以在打开完成后看到以下缓存状态：

```
[a: 0 (v=1,d=0,r=1)]
```

这意味着文件 'a' 在缓存中，值为 '0'，并具有三个关联的状态位：v（有效）、d（脏）和 r（引用计数）。有效位跟踪内容是否有效；现在是，因为缓存尚未被回调无效。脏位在文件被写入且必须在关闭时刷新回服务器时改变。最后，引用计数跟踪文件被打开了多少次（但尚未关闭）；这用于确保客户端获取文件的旧值，直到所有读者都关闭了文件然后重新打开。

完整选项列表如下：

```
选项:
  -h, --help            显示此帮助信息并退出
  -s SEED, --seed=SEED  随机种子
  -C NUMCLIENTS, --clients=NUMCLIENTS
                        客户端数量
  -n NUMSTEPS, --numsteps=NUMSTEPS
                        每个客户端将执行的操作数量
  -f NUMFILES, --numfiles=NUMFILES
                        服务器中的文件数量
  -r READRATIO, --readratio=READRATIO
                        读取/写入比例
  -A ACTIONS, --actions=ACTIONS
                        精确指定客户端操作，例如，
                        oa1:r1:c1,oa1:w1:c1 指定两个客户端；每个打开文件 a，
                        客户端 0 读取它而客户端 1 写入它，然后各自关闭它
  -S SCHEDULE, --schedule=SCHEDULE
                        要运行的确切调度；01 在客户端 0 和 1 之间轮转调度。
                        未指定将导致随机调度
  -p, --printstats      打印额外统计信息
  -c, --compute         为我计算答案
  -d DETAIL, --detail=DETAIL
                        给出答案时的详细级别 (1:服务器操作,2:失效,
                        4:客户端缓存,8:额外标签)；组合使用多个选项
```
