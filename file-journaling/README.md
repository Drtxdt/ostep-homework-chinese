# 概述

本作业介绍了 `fsck.py`，一个简单的文件系统模拟器。对 `vsfs.py`模拟器（在之前关于文件系统实现的章节中描述过）有一定了解会有所帮助，但不是必需的。

该工具会改变文件系统的磁盘状态，然后留给你（用户）来检测问题，这很像一个文件系统一致性检查器（例如 fsck）会做的事情。文件系统本身基于一个简化的 VSFS（在之前章节描述的非常简单的文件系统）。该工具首先生成一个磁盘上的文件系统；然后，随机地或通过更具体的控制，工具会改变文件系统某个方面的磁盘状态。

你面临的挑战是找出文件系统磁盘状态的哪一部分被改变并因此变得不一致。在许多情况下，问题应该很容易检测到；在其他情况下，可能不那么容易。

让我们深入细节。以下是该工具的一个运行示例：

```
prompt> ./fsck.py -S 1
...
文件系统的最终状态：

inode 位图 1100100010000010
inodes       [d a:0 r:4] [f a:-1 r:1] [] [] [d a:8 r:2] [] [] [] 
             [d a:6 r:2] [] [] [] [] [f a:15 r:1] [f a:12 r:1] []
数据位图  1000001010001001
数据         [(.,0) (..,0) (g,8) (t,14) (w,4) (m,13)] [] [] [] [] [] 
             [(.,8) (..,0)] [] [(.,4) (..,0) (p,1)] [] [] [] [z] [] [] [g]

你能找出文件系统是如何损坏的吗？

prompt>
```

在这种情况下，我们运行该工具时设置了标志 `-S`为 1（原因稍后会清楚）。要理解磁盘状态的改变以及可能的不一致，首先你必须理解文件系统的格式。它（正如承诺的那样！）非常简单。

首先，有一个 inode 位图，它标记每个对应的 inode 是已分配 (1) 还是空闲 (0)；在这种情况下，有 16 个 inode，其中 inode 0, 1, 5, 9, 和 14 被标记为已分配。

其次，有 16 个 inode。未分配的 inode 显示为空括号 —— `[]`—— 而已分配的 inode 有如上所示的 `[f a:15 r:1]`等内容。每个已分配的 inode 有三个字段。第一个是文件类型，'d' 表示目录，'f' 表示常规文件。第二个是一个地址字段 'a'，它要么指向单个数据块（在这个受限的文件系统中，文件/目录只能包含一个数据块），要么是 -1 表示文件为空。第三个是引用计数；对于常规文件，这代表该文件在文件系统中被链接的次数；对于目录，它代表该目录内的目录数量。

第三是数据位图。很像 inode 位图，它指示文件系统中每个数据块是已分配还是空闲。

最后，是数据块本身。对于常规文件，数据块内容是任意的（但通常包含简单的内容，比如一个字母；在现实生活中，这些只包含文件数据块），而目录内容则结构化为 (名称, inode 编号) 对的列表。

这个文件系统中的根目录总是 inode 编号 0；你可以从这一点开始来构建对整个文件系统及其内容的了解。

这个特定的文件系统包含以下文件和目录：

```
目录: '/', '/g', '/w'
  文件: '/t', '/m', '/w/p'
```

你能看出为什么吗？花些时间，从根目录开始，看看你能否弄清楚。如果不能，那么这份作业对你来说会很困难。

现在，我们来谈谈不一致的部分。这个特定的文件系统内部有一个简单具体的不一致。你能看出来吗？

一旦你花了一些时间思考，运行带有 `-c`标志的工具来检查你是否正确：

```
prompt> ./fsck.py -S 1 -c

文件系统的初始状态：

inode 位图 1100100010000110
inodes       [d a:0 r:4] [f a:-1 r:1] [] [] [d a:8 r:2] [] [] [] [d a:6 r:2]
             [] [] [] [] [f a:15 r:1] [f a:12 r:1] []
数据位图  1000001010001001
数据         [(.,0) (..,0) (g,8) (t,14) (w,4) (m,13)] [] [] [] [] [] 
             [(.,8) (..,0)] [] [(.,4) (..,0) (p,1)] [] [] [] [z] [] [] [g]

损坏::INODE BITMAP 损坏的位 13

文件系统的最终状态：

inode 位图 1100100010000010
inodes       [d a:0 r:4] [f a:-1 r:1] [] [] [d a:8 r:2] [] [] [] [d a:6 r:2]
             [] [] [] [] [f a:15 r:1] [f a:12 r:1] []
数据位图  1000001010001001
数据         [(.,0) (..,0) (g,8) (t,14) (w,4) (m,13)] [] [] [] [] [] 
             [(.,8) (..,0)] [] [(.,4) (..,0) (p,1)] [] [] [] [z] [] [] [g]

prompt>
```

从输出中你可以看到，inode 位图的第 13 位被改变了，将该位标记为空闲，而实际上该 inode 是已分配的。你可以通过查看 inode 13 本身来看出不一致，它看起来像这样：`[f a:15 r:1]`（一个常规文件，指向数据块 15）。你还知道这个文件在根目录中 (/m)。因此，推断出这里的 inode 位图发生了变化是相当简单的。

许多其他损坏是可能的（但该工具一次只引入一个状态改变，以使你的生活简单）。完成作业以了解更多关于它们的信息。

该工具可用的其他标志有：

```
prompt> ./fsck.py -h
选项:
  -h, --help            显示此帮助信息并退出
  -s SEED, --seed=SEED  第一个随机种子（用于文件系统）
  -S SEEDCORRUPT, --seedCorrupt=SEEDCORRUPT
                        第二个随机种子（用于损坏）
  -i NUMINODES, --numInodes=NUMINODES
                        文件系统中的 inode 数量
  -d NUMDATA, --numData=NUMDATA
                        文件系统中的数据块数量
  -n NUMREQUESTS, --numRequests=NUMREQUESTS
                        要模拟的请求数量
  -p, --printFinal      打印最终的文件/目录集
  -w WHICHCORRUPT, --whichCorrupt=WHICHCORRUPT
                        执行特定的损坏
  -c, --compute         为我计算答案
  -D, --dontCorrupt     不实际损坏文件系统
```

有两个随机种子的原因值得描述。第一个 `-s`，创建不同的随机文件系统。第二个 `-S`，插入不同的文件系统状态改变（损坏）。因此，你可以生成一个特定的随机文件系统，然后对其尝试一堆不同的随机损坏。

其他一些标志只控制文件系统的大小（`-i`和 `-d`），而另一些则决定对文件系统运行多少次文件操作（`-n`）。`-p`标志打印（未损坏的）文件系统的内容；`-w`标志允许你指定特定的损坏（尽管这主要用于测试）；`-D`标志关闭损坏，允许你检查一个完整的文件系统。
